# AI 辅助开发心得 — 以任务看板项目为例

## 项目背景

本项目是一个基于 Spring Boot 的任务看板应用，采用 Thymeleaf + MyBatis-Plus + Sa-Token 技术栈，实现了任务的增删改查、拖拽状态变更、用户认证等功能。整个开发过程深度借助 Claude Code（AI 编程助手）完成，以下是在此过程中积累的 AI 使用心得与经验总结。

---

## 一、Skill 文件：让 AI 成为"懂规矩的队友"

### 1.1 为什么需要 Skill 文件

AI 编程助手的能力很强，但它不了解你的项目偏好。如果不加约束，AI 可能会：

- 使用 JPA 而你偏好 MyBatis-Plus
- 引入 Spring Security 而你已选定 Sa-Token
- 采用 Controller → Service → Mapper 的简单分层，而你需要 Facade 层做业务编排
- 生成不符合团队规范的代码风格

**Skill 文件的本质是一份"团队开发规范"**，只不过读者从人类变成了 AI。

### 1.2 Skill 文件的核心内容

本项目的 Skill 文件（`.claude/skills/springboot-dev-standards.md`）约 500 行，涵盖了：

| 类别 | 具体内容 |
|---|---|
| **技术栈约束** | 明确指定每个技术选型，并用"禁止"标注不允许的替代方案 |
| **分层架构** | Controller → Facade → Service → Mapper 的严格分层，禁止跨层调用 |
| **代码风格** | Lombok 注解规范、构造器注入方式、命名约定 |
| **注释规范** | JavaDoc 必须写、注释说明"为什么"而非"怎么做" |
| **查询策略** | 简单查询用 LambdaQueryWrapper，复杂查询用 XML Mapper |
| **DTO/VO 规范** | 命名规则、转换方式、禁止 Entity 直接暴露 |

### 1.3 实践心得

- **越具体越好**：不要写"使用合适的 ORM"，而要写"MyBatis-Plus 3.5.5，禁止使用 JPA/Hibernate"。AI 需要明确的边界。
- **给出代码示例**：每个规范都附带"正确示例"和"错误示例"，AI 能更准确地理解意图。
- **迭代优化**：Skill 文件不是一次写好的。项目经历了 5 次提交，其中有 2 次是专门优化 Skill 文件，说明它需要在实践中不断打磨。

## 二、Prompt 工程：如何与 AI 高效沟通

### 2.1 任务拆解

将复杂需求拆解为明确的步骤，比直接说"帮我做一个任务看板"效果好得多：

```
差的 Prompt：帮我做一个任务管理系统
好的 Prompt：
1. 创建 Todo 实体类，包含 id、title、description、status、userId、createdAt、updatedAt 字段
2. 创建 TaskStatus 枚举，包含 TODO、IN_PROGRESS、DONE 三个状态
3. 创建 TodoMapper 接口继承 BaseMapper
```

### 2.2 上下文管理

- **先让 AI 阅读 Skill 文件**：在对话开始时，AI 会自动加载 `.claude/skills/` 下的规范文件，确保后续生成的代码符合约定。
- **引用具体文件**：说"参考 TodoFacade 的写法"比说"按照之前的风格"更有效。
- **及时纠偏**：当 AI 生成的代码不符合预期时，立即指出并说明原因，AI 会在后续交互中记住纠正方向。

### 2.3 渐进式开发

不要期望 AI 一次性生成完美的完整项目。更实际的流程是：

```
需求分析 → 生成骨架代码 → 逐层填充 → 调整细节 → 测试验证 → 迭代优化
```

每一步都可以让 AI 参与，但人类需要把控方向和质量。

## 三、AI 擅长的场景

### 3.1 样板代码生成

分层架构中存在大量结构相似的代码（Entity、Mapper、Service 接口、Service 实现、DTO、VO），AI 可以在几秒内完成这些重复性工作，且保持风格一致。

### 3.2 规范执行

一旦 Skill 文件定义清楚，AI 生成的代码会严格遵循：
- 所有 Service 类都使用 `@RequiredArgsConstructor` 注入
- 所有 Facade 类都包含 Entity → VO 的转换方法
- 所有查询都使用 `LambdaQueryWrapper` 而非手写 SQL

### 3.3 前端模板生成

Thymeleaf 模板 + CSS 样式 + JavaScript 交互逻辑，AI 可以一次性生成包括拖拽功能在内的完整前端代码。

### 3.4 配置与集成

pom.xml 依赖配置、application.properties 参数、Sa-Token 拦截器配置等，AI 能快速生成且减少配置遗漏。

## 四、AI 的局限与人的角色

### 4.1 架构决策仍需人来把控

AI 可以执行你定义的架构规范，但"为什么选择 Facade 模式而非直接 Service 调用""为什么用 Sa-Token 而非 Spring Security"这类决策需要人来做。

### 4.2 文档同步容易被忽略

本项目的一个教训：**README、USAGE.md、PROJECT_SUMMARY.md 中的技术栈描述与实际 pom.xml 不一致**（文档写的 Spring Boot 2.7.0 + H2，实际是 3.2.5 + MySQL）。AI 在升级代码后不会自动更新所有文档，这需要人来协调或明确要求。

### 4.3 安全性需要人来审查

AI 生成的认证代码使用了明文密码比对（`password.equals(user.getPassword())`），没有引入密码哈希。对于安全相关的实现，人必须介入审查。

### 4.4 测试覆盖需要主动要求

AI 不会自动生成完整的测试用例。如果需要测试覆盖，必须明确要求："为 TodoFacade 编写单元测试，覆盖创建、查询、更新、删除、状态变更等场景"。

## 五、效率对比

| 维度 | 纯人工开发 | AI 辅助开发 |
|---|---|---|
| 项目骨架搭建 | 30-60 分钟 | 5-10 分钟 |
| CRUD 代码编写 | 2-4 小时 | 30-60 分钟 |
| 前端页面开发 | 2-3 小时 | 30-45 分钟 |
| 配置调试 | 1-2 小时 | 15-30 分钟 |
| 代码规范一致性 | 依赖人工 review | Skill 文件保证一致性 |
| **总计** | **约 1-2 天** | **约 2-4 小时** |

> 注：以上为个人体感估算，效率提升约 3-5 倍，主要节省在样板代码和重复性工作上。

## 六、最佳实践总结

### 6.1 建立 Skill 文件

- 将团队技术栈、编码规范、架构约定写成 Skill 文件
- 包含具体的代码示例（正确 vs 错误）
- 在实践中持续迭代优化

### 6.2 善用分层架构

- 让 AI 严格遵循分层调用关系
- Facade 层作为业务编排的枢纽，有效隔离 Controller 和 Service
- DTO/VO 的使用让数据流向清晰可控

### 6.3 保持人类主导

- **AI 是执行者，人是决策者**：架构选型、安全设计、业务逻辑的合理性需要人来判断
- **AI 生成，人来审查**：特别是涉及安全、数据一致性、异常处理的代码
- **AI 加速，人来验收**：最终的功能正确性需要人工测试验证

### 6.4 管理好上下文

- 对话开始时确保 Skill 文件被加载
- 复杂任务分步骤进行，每步确认后再继续
- 发现偏差及时纠正，避免错误累积

---

## 七、结语

AI 编程助手不是"自动写代码的机器"，而是一个**高效率的协作者**。它的价值在于：

1. **消除重复劳动**：样板代码、配置文件、CRUD 逻辑的快速生成
2. **保证一致性**：通过 Skill 文件，让每一行代码都符合团队规范
3. **降低心智负担**：开发者可以专注于业务逻辑和架构设计，而非记忆 API 细节

但最终，**代码的质量取决于使用者的判断力**。AI 放大了开发者的能力，但不能替代开发者对业务的理解、对架构的思考、对安全的敏感。

用好 AI 的关键不在于 Prompt 写得多花哨，而在于**你是否清楚自己要什么**。
